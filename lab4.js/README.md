## Реализация

### Шаг 1 — Класс `Item`

Базовый класс, представляющий предмет в инвентаре.

| Поле | Тип | Описание |
|------|-----|----------|
| `name` | `string` | Название предмета |
| `weight` | `number` | Вес (кг) |
| `rarity` | `string` | Редкость: `common`, `uncommon`, `rare`, `legendary` |

**Методы:**
- `getInfo()` — возвращает строку с описанием предмета
- `setWeight(newWeight)` — изменяет вес; игнорирует значения ≤ 0

```js
const sword = new Item("Steel Sword", 3.5, "rare");
console.log(sword.getInfo());
// [RARE] Steel Sword | Вес: 3.5 кг

sword.setWeight(4.0);
// Вес "Steel Sword" обновлён до 4 кг
```

### Шаг 2 — Класс `Weapon` (расширяет `Item`)

| Поле | Тип | Описание |
|------|-----|----------|
| `damage` | `number` | Урон оружия |
| `durability` | `number` | Прочность (0–100), автоматически зажимается функцией `clamp` |

**Методы:**
- `getInfo()` — расширяет родительский метод, добавляя урон и прочность
- `use()` — уменьшает прочность на 10; если прочность уже 0, выводит предупреждение
- `repair()` — восстанавливает прочность до 100

```js
const bow = new Weapon("Longbow", 2.0, "uncommon", 15, 100);
bow.use();   // Прочность: 90/100
bow.repair(); // Прочность: 100/100
```

### Шаг 4А — Опциональная цепочка (`?.`)

Механизм `?.` позволяет безопасно обращаться к свойствам объектов, которые могут быть `null` или `undefined`, не вызывая `TypeError`.

```js
function printItemInfo(item) {
  const info = item?.getInfo() ?? "Предмет не найден";
  console.log(info);
}

printItemInfo(sword);     // [RARE] Steel Sword | Вес: 4 кг
printItemInfo(null);      // Предмет не найден
printItemInfo(undefined); // Предмет не найден
```

Также применяется при итерации по массиву с возможными пустыми слотами:

```js
const items = [sword, bow, null, legendaryAxe];
items.forEach((item, i) => {
  console.log(`Слот ${i + 1}: ${item?.name ?? "(пусто)"}`);
});
```

### Шаг 4Б — Функции-конструкторы (ES5)

Классы `Item` и `Weapon` переписаны с использованием функций-конструкторов и `prototype`.

**Наследование через `Object.create`:**

```js
WeaponConstructor.prototype = Object.create(ItemConstructor.prototype);
WeaponConstructor.prototype.constructor = WeaponConstructor;
```

**Вызов родительского конструктора:**

```js
function WeaponConstructor(name, weight, rarity, damage, durability) {
  ItemConstructor.call(this, name, weight, rarity); // как super()
  this.damage = damage;
  this.durability = clamp(durability, 0, 100);
}
```

Проверка цепочки прототипов:

```
dagger instanceof WeaponConstructor → true
dagger instanceof ItemConstructor   → true
bow instanceof Weapon               → true
bow instanceof Item                 → true
```

## Контрольные вопросы

**1. Какое значение имеет `this` в методах класса?**

В методах класса `this` указывает на экземпляр объекта, для которого вызван метод. При вызове `sword.getInfo()` внутри метода `this` будет ссылаться на объект `sword`. Важно помнить, что значение `this` зависит от контекста вызова: при передаче метода в колбэк без привязки `this` может стать `undefined` (в строгом режиме) или `window`/`global`.

**2. Как работает модификатор доступа `#` в JavaScript?**

Поля, объявленные с префиксом `#`, являются **приватными** и доступны только внутри тела класса. Попытка обратиться к ним снаружи вызывает `SyntaxError`. В отличие от соглашения об именовании с `_`, приватность через `#` обеспечивается на уровне языка и не может быть обойдена. Пример: `this.#durability = 100;`.

**3. В чём разница между классами и функциями-конструкторами?**

| | `class` | Функция-конструктор |
|--|---------|---------------------|
| Синтаксис | ES6+, более читаемый | ES5, более многословный |
| Поднятие (hoisting) | Не поднимаются | Поднимаются как функции |
| Строгий режим | Всегда `"use strict"` | Только явно |
| Наследование | `extends` / `super` | `Object.create` / `.call` |
| Статические методы | `static` ключевое слово | Присваивается напрямую функции |

По сути, `class` — это синтаксический сахар над прототипным наследованием. Оба подхода используют одну и ту же систему прототипов под капотом.
